# What technologies are used?
This page is intentionally a design-decisions summary, not a dependency dump. Each repo uses a different stack because each one is solving a different operational problem: desktop mission planning, field operation, protocol bridging, and onboard autonomy all have different constraints.

| Repo | Primary technologies | Why this stack fits |
| --- | --- | --- |
| `Tower` | Electron, React, TypeScript, Zustand, MapLibre, deck.gl | Desktop mission planner with a hybrid MapLibre/deck.gl renderer built for dense 3D editing on GPU-equipped hardware |
| `Tower-Field` | React, TypeScript, Vite, PWA, Leaflet, Leaflet.draw | Browser-first field interface with swappable basemaps, transparent overlay deployment, and touch-first operation on low-compute devices |
| `tower-server` | Go, protobuf, UDP multicast, WebSocket, JSON | Protocol authority that turns lossy vehicle telemetry into validated UI sessions, fleet state, and command control |
| `maf_aerial` | Pure C++, BehaviorTree.CPP, MAVLink, protobuf, MCAP, Docker | Single-process onboard autonomy runtime with direct control paths, boundary-only serialization, and config-driven deployment from SITL to VOXL |

## Tower

[Tower](https://github.com/EthanMBoos/Tower) is the heavyweight desktop operator interface. It is built for an **indoor, power-user workflow** that favors information density, advanced configuration, and GPU-backed mapping over beginner-friendly simplicity. That is why the project uses **Electron** for a cross-platform desktop shell, **React 19 + TypeScript** for complex UI composition, **Zustand** for low-boilerplate shared state, and **Vite** for a fast build loop.

The map stack is the real differentiator: **MapLibre GL JS** owns the base map, style swaps, DEM terrain, and the features that must drape cleanly on terrain, while **deck.gl** is injected directly into MapLibre's render loop through **`MapboxOverlay` in interleaved mode**, so both systems share one WebGL context instead of fighting through separate canvases. In practice, that creates a deliberate split renderer: **EditableGeoJsonLayer**, **SolidPolygonLayer**, **PathLayer**, and **ScenegraphLayer** handle interactive editing, 3D zone walls and ceilings, air routes, and airborne vehicle models, while native MapLibre symbol and line layers are reserved for ground vehicles, ground routes, search points, and pins because MapLibre can GPU-drape those reliably onto terrain and deck.gl cannot. Features that need explicit elevation are sampled asynchronously with `queryTerrainElevation()` and batched with `requestIdleCallback`, then rendered with different depth settings depending on whether they need to sit on top of terrain or participate in full 3D occlusion. That architecture is intentionally compute-heavy, but it buys the desktop app the dense, high-fidelity mission editing pipeline it is optimized for.

## Tower-Field

[Tower-Field](https://github.com/EthanMBoos/Tower-Field) exists because the field environment has almost the opposite constraints. It is an **outdoor operator tool** for high-daylight visibility, touch use, low-end hardware, and minimal cognitive load. For that reason it stays in the browser as a **Progressive Web App** built with **React + TypeScript + Vite** instead of becoming another desktop or mobile-native application.

The biggest technology decision is using **Leaflet** with **react-leaflet** and **Leaflet.draw** rather than MapLibre or deck.gl, but the deeper architectural point is that the field app treats the basemap as a replaceable backend instead of the center of the system. In the current implementation the map is just a `MapContainer` plus a `TileLayer` for imagery, with mission geometry, polylines, markers, and draw state layered above it through React and Leaflet primitives like **`FeatureGroup`**, **`Polyline`**, custom **`divIcon`** vehicle markers, and a programmatic **DrawController** that turns app state into `L.Draw` handlers. Because the basemap is only a tile source, it can be swapped to different imagery providers, run standalone, or disappear entirely when the app is embedded as a **transparent WebView overlay** on top of TAK, WinTAK, ATAK, or another C2 system. That is a major selling point beyond raw performance: operators can keep their existing situational-awareness picture and add mission planning, vehicle control, and lightweight overlays without adopting an entirely new mapping stack. The browser-first packaging matters for the same reason. A **PWA** avoids native bridge overhead, app-store deployment, and GPU assumptions, while keeping WebSocket transport native to the browser and making the same app usable as a standalone field tool, a tablet shortcut, a side-by-side browser tab, or an embedded overlay. The implementation also leans into field ergonomics, with high-contrast styling, large touch targets, simplified draw modes, and minimal rendering complexity, so the repo is not just lighter than Tower; it is structurally more flexible about where and how it can be deployed.

## tower-server

[tower-server](https://github.com/EthanMBoos/tower-server) is the protocol and transport bridge between vehicles and operator UIs, so its stack is driven by deployment reliability rather than UI concerns. The project uses **Go** with **protobuf**, **UDP multicast**, **WebSocket**, and **JSON**. Vehicles speak compact protobuf messages over multicast, while the UI gets browser-friendly JSON over a WebSocket connection.

**Go** fits this repo because tower-server is doing much more than passing bytes through. It is the authoritative boundary where lossy, vehicle-facing **protobuf over UDP multicast** becomes validated, deduplicated, browser-facing **JSON over WebSocket**. In the implementation, raw datagrams enter through one or more multicast listeners, are size-checked and unmarshaled in `DecodeVehicleMessage()`, translated into JSON frames, stamped with an authoritative server timestamp, and then pushed through a central registry that owns vehicle discovery, capability tracking, and the `online → standby → offline` state machine. That same boundary is also where sequence-number deduplication happens, including wrap-around handling and explicit tracker resets when a vehicle goes offline and later comes back, which is critical for rebooted robots whose sequence counters restart at zero. On the UI side, the WebSocket handshake is not just a socket upgrade: `hello`/`welcome` establishes the entire client session with a fleet snapshot, negotiated protocol version, registered extension namespaces, and loaded YAML manifests, making tower-server the source of truth for what the UI is allowed to render and command.

That is why the language choice matters less in terms of raw speed and more in terms of operational shape. Go makes it easy to run this as a single binary with fast startup, simple cross-compilation, and straightforward concurrency for the fan-in and fan-out problem: many multicast telemetry sources in, many UI clients out. The code also uses that concurrency model to separate concerns cleanly: the command router validates the target vehicle and its advertised capabilities, rate-limits commands per vehicle, encodes core and extension commands back into protobuf, multicasts them on a separate outbound group, and tracks pending commands so tower-server can emit immediate accepts, explicit rejections, or synthetic timeout acknowledgments if the vehicle never answers. That combination is the real selling point of tower-server: it is the protocol authority, extension registry, liveness monitor, command admission controller, and browser compatibility layer for the whole platform. A simpler language/runtime with strong built-in networking and observability support is exactly what that job wants.

## maf_aerial

[MAF](https://github.com/EthanMBoos/maf_aerial) is an onboard autonomy runtime, so its constraints are much closer to control loops, behavior execution, and hardware integration than to UI or protocol bridging. Even in its current design-heavy phase, it is organized as a **single-process, pure C++ autonomy runtime** where serialization happens only at external boundaries, not between internal components. That is the central architectural decision. Instead of ROS 2 nodes, DDS discovery, callback queues, and repeated serialization hops, the system is built around direct C++ classes, shared state guarded by `std::shared_mutex`, lock-free queues where high-rate producer and consumer handoff matters, and synchronous function calls when components live in the same control path. The internal data flow is meant to stay close to the machine: PX4 state arrives, behavior logic evaluates mission and failsafe conditions, planners and controllers compute outputs on local snapshots, and setpoints go straight back to the autopilot without middleware sitting in the middle.

That is why the stack is such a specific mix. **BehaviorTree.CPP** is not just a behavior library here; it is the mission executive that sequences nodes like takeoff, hover, waypoint following, landing, heartbeat checks, and battery failsafes in one reactive tree. **mavlink-c** talks directly to PX4 over UDP in SITL and serial on hardware, which matters because MAF is designed around offboard control, continuous setpoint streaming, and tight control over what gets sent to the flight stack. **Protobuf** is reserved for the tower-server boundary, where `AerialCommand` and `AerialTelemetry` become extension payloads for the wider Tower system. **MCAP** is used for flight recording so the project keeps rosbag-style inspection and Foxglove compatibility without taking on ROS itself. **yaml-cpp** and a layered config hierarchy are equally important: the same binary is intended to move between PX4 SITL and ModalAI Starling hardware by swapping config layers for defaults, domain, platform, and vehicle, rather than by forking codepaths. On top of that, **CMake**, **Docker Compose**, multi-arch container builds, and VOXL-specific deployment scaffolding turn the architecture into something that can actually live on constrained edge hardware. The real selling point is not just determinism. It is that MAF is trying to preserve the clarity and directness of production autonomy software while still keeping simulation, field deployment, logging, and C2 integration in one coherent system.

## Why the stacks differ

Across the repos, the pattern is consistent: the technology follows the operator, hardware, and deployment environment.

- The desktop planner uses the richest stack that best supports dense 3D mission planning.
- The field app uses the lightest stack that still supports operational overlays and touch interaction.
- tower-server uses the easiest stack to ship as a reliable cross-platform bridge.
- The autonomy framework uses the most direct stack for control, timing, and hardware integration.

That separation is intentional. The question here is not which technology stack is best in the abstract. It is which stack best fits the real constraints of each repo: trained indoor operators, field operators on edge devices, protocol translation on shared networks, or onboard autonomy running close to the aircraft. Trying to force all of those jobs into one shared technology choice would make every repo worse at its actual job.
